How Does control-efficacy analytics Work?

Quick walkthrough — 90 days of incidents, the three answers leadership wants, in 2 minutes.
📥

1. Ingest

Every four hours, tickets are pulled from XSOAR across 17 active detection sources (CrowdStrike, QRadar, Splunk, Vectra, Tanium, and more) and cached locally. Each ticket is enriched with derived fields: security category, root cause, attack vector, disposition, and the owning technology team.

📊

2. Aggregate (90-day rolling window)

On a biweekly cadence (Mondays at 6 AM ET), the scheduler rolls up the last 90 days of enriched tickets into four headline KPIs:

  • Total Incidents — volume across every source
  • Control Effectiveness — % blocked automatically vs. escalated to a human
  • Malicious True Positives — confirmed real attacks after human review
  • Top Attack Vector — the #1 vector by alert share
📈

3. Visualise

Five interactive charts tell the story the KPIs summarise:

  • Category Impact Heatmap — security category × impact type
  • Control Effectiveness Dashboard — blocked vs. escalated per source, plus a donut of the overall split
  • Root Cause × Detection Source — stacked bars showing which sources surface which root causes
  • Awareness Trends — weekly line chart of awareness-related incidents, with spike weeks called out
  • Repeat Offenders (Pareto) — which users and hosts generate the long tail of noise
🤖

4. Control-Efficacy Analysis

The analysis layer answers three questions: are our controls holding?, where are we getting hit?, and what should we fix next? It is not an incident summarizer.

  • Control Interaction Records (CIRs) — every ticket is deterministically tagged with an expected control and an outcome: BLOCKED, ALERTED, MISSED, BYPASSED, NOT_APPLICABLE, or UNKNOWN, with a confidence level and machine-readable evidence.
  • Failure Modes — the LLM clusters non-BLOCKED CIRs into named failure mechanisms (e.g. "MFA bypass via helpdesk reset") rather than category labels. Each has supporting incident IDs, hypothesized root causes, and confirmation steps.
  • Fix Candidates (FPI-ranked) — each fix maps to at least one failure mode and is scored on five axes: Recurrence, Impact, Time penalty, Human cost, and External intel alignment. FPI = 2R + 2I + 1.5T + 1.5H + 0.5E. Every fix ships with success metrics and a validation plan.
  • Human Compensation Signals — process-debt indicators (repeat triage, duplicates, long containment cycles) that point at missing telemetry or tooling, never at individuals.
  • Data Gaps — fields the analysis wanted but could not compute (e.g. per-incident dwell time). Audit-defensible by design.

Operating rules enforced on every run: no people-performance evaluation, no fabricated counts or dates, FACT vs INFERENCE vs UNKNOWN labeling, and external intel can only weight an internally observed failure mode — never serve as proof of exposure.

📖

5. Data Story Mode

A guided five-step narrative (Landscape → Shield → Threats → Patterns → Cost) walks a board-level audience through the posture end-to-end, with optional audio narration and spotlight mode. Built so leadership can self-serve the story without a briefing deck.

⚙️

Configuration

Refresh cadence: Biweekly, Mondays at 6 AM ET (scheduled via ir-scheduler)
Rolling window: 90 days
Detection sources: 17 active (XSOAR-integrated)
Analysis split: CIRs and run metadata are computed deterministically in Python; the LLM only produces judgment-heavy sections (failure modes, fixes, signals, gaps, quality checks).
AI model: the LLM, structured output (Pydantic-validated) with explicit enumerations and confidence ratings.
Storage: Full control-efficacy JSON (ai_analysis) plus derived markdown, written alongside KPIs under web/static/charts/.
Fallback: If the LLM is unreachable, the previous run's analysis is carried forward so leadership always sees current-state evidence.
