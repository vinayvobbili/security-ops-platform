🛡️ How the Code Security Scanner works

An agentic source-code audit that splits the work into two passes — a wide-net navigation pass and a skeptical validation pass — the way modern AI security tooling finds and then validates bugs.

1️⃣ Map

The repo is walked once (deterministically) to build a code map — languages, dependency manifests, likely entrypoints, and the list of source files worth auditing. No LLM, no guessing.

2️⃣ Navigate — LLM tool loop

An LLM drives a read-only tool loop (list / read / grep) scoped to the repo. It finds where untrusted input enters, follows it toward dangerous sinks (SQL, command, template, file paths, SSRF, deserializers, auth checks), and records candidate findings. It casts a wide net on purpose.

3️⃣ Validate — second LLM pass (refute-first)

Every candidate is re-checked against the real code slice by a second LLM pass, prompted to refute it: does untrusted input actually reach the sink unsanitized? Candidates it can't stand up are culled (and shown, never silently dropped) — this is what kills the false-positive flood.

4️⃣ Report

What survives is a ranked list of findings with severity, a confidence score, the source→sink flow, repro steps, and a recommended fix. Everything is read-only — no writes, no patches, no merge requests (that's a later slice). Every scan is logged with an audit trail.

🔒 Safety

Scanning never executes the target's code, and each scan runs inside an isolated container — non-root, read-only, no network, with only the target mounted. A git URL is cloned in a separate throwaway container (no host access), hardened against malicious-repo tricks (no submodules, no repo hooks, transport allowlist). Secret-shaped files (keys, certs, .env, credential stores) are never read, and local scans are confined to allowed roots.
