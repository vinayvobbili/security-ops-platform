How does Vulnerability Deep Dive work?

▶ 3-minute walkthrough · every tab

What it is

An automated first-pass triage of the vulnerability war-room's CVE backlog — 3,000+ CVEs — replicating, at scale, the manual "look up each CVE and decide if remediation is needed" loop an analyst would otherwise run by hand.

Watch the short walkthrough, or read the steps below for how each verdict is reached.

🧩

Step 1 — gather the facts

It starts from the spreadsheets the war room already shares — the CVE impact-count list and the Prisma container-scan export — then enriches each CVE with the same things an analyst would look up by hand:

  • NVD — primary CVE record (CVSS, affected products).
  • CVE.org — fallback that catches brand-new CVEs still in NVD's backlog.
  • Veracode SCA — which of our applications actually carry the affected component.
  • CISA KEV + EPSS — is it being exploited in the wild right now, and how likely is exploitation.
  • Enterprise App Inventory (EAI) — which carrying applications are internet-facing, and how business-critical they are.
🥊

Step 2 — two analysts debate it

Instead of one opinion, the facts go to two AI analysts who argue opposite sides — the same adversarial check a good bug-hunting team runs:

  • 🔴 an attacker — assumes the flaw is reachable and hunts the realistic exploit path;
  • 🔵 a skeptic — assumes it's noise until proven exploitable in our fleet, and picks the preconditions apart.

Each scores urgency 0–100. Pitting the two against each other weeds out false positives far better than a single voice — and the disagreements that survive are exactly the ones worth a human's time.

⚖️

Step 3 — reconcile, or flag for review

When the two agree, the call is made automatically. When they disagree hard on a serious CVE — one sees a real threat, the other sees noise — it's tagged REVIEW Needs Review for a human to settle, rather than being silently waved through or thrown away. Everything else auto-resolves toward the cautious read so the queue stays small.

🚦

Priority (P1–P4)

A composite urgency score — deliberately not CVSS alone. It blends CVSS with EPSS (probability of exploitation), CISA KEV (known-exploited), and pre-auth (no credentials needed).

  • P1 — 72h · P2 — 30d · P3 — 90d · P4 — 180d (remediation SLAs).
🔒

MUST-ACT — the non-negotiables

A short list of factor combinations means we act, period — these force P1 regardless of what the debate concludes or how the reachability cap would otherwise lower it. A MUST-ACT badge shows which rule fired:

  • known-exploited (KEV) + internet-facing
  • known-exploited (KEV) + pre-auth
  • internet-facing + pre-auth + confirmed-reachable
  • CVSS ≥ 9.8 + internet-facing
📍

Reachability — and how it caps priority

An honest evidence-level presence signal — not yet a proven runtime code-path determination (that needs container introspection, which isn't built yet):

  • present-confirmed — Veracode confirms it's in our portfolio.
  • present-conditional — present, but gated by limiting preconditions.
  • pending-image — base-image OS package we can't confirm without pulling the container image.
  • unknown — no presence signal.

Reachability caps the final priority. The base tier (KEV / EPSS / pre-auth) is then bounded by a ceiling that combines this code-path reachability with internet exposure (the EAI internet-facing flag): internet-facing + reachable can be P1; a base-image or internal-only finding caps lower. So a CVSS 9.8 in a base image with no reachable code path is not a P1 emergency — the tool now says so. The container composite is likewise multiplied by a reachability factor (app-layer ×1.0, mixed ×0.7, base-image-only ×0.4).

⚡

Where the leverage is

CVEs are clustered by component so one decision — e.g. upgrade Tomcat — retires many CVEs at once. Start at the top of that list to clear the most findings for the least effort.

⚠️
This is an automated first pass meant to accelerate analyst review, not replace it — the two-analyst debate triages the backlog and surfaces the genuine disagreements, but a human still owns the Needs Review queue and the final call. Reachability is an evidence-level signal (Veracode presence + base-image attribution + EAI exposure), not a proven runtime code-path. Treat verdicts as a sharp starting point, not the last word.
