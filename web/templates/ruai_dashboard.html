How Does RUAI Screening Work?

Quick walkthrough — the reviewer's workflow in under 3 minutes.
📋

1. Submission

A submitter fills out the screening form (or uploads an Excel survey + supporting docs). The form captures the AI use case description, business purpose, data sensitivity, vendor details, and whether the AI makes or influences decisions.

🤖

2. AI Security Review

An LLM (the LLM) automatically reviews the submission. It evaluates 6 threat boundaries (User, Application, Model, Tool/Agent, Data, Infrastructure) and 6 AI-specific threats (Prompt Injection, Data Poisoning, Model Inversion, Hallucination, Model Misuse, Autonomous Actions). It checks security controls against each boundary and rates them as Present, Partial, or Missing.

📊

3. Risk Assessment Output

The AI produces a structured JSON report containing:

  • Overall risk score — Low / Medium / High / Critical
  • Completeness issues — unanswered or vague responses
  • Risk flags — specific concerns by security domain
  • Threat boundary analysis — per-boundary components, risks, gaps
  • Control coverage map — which controls are present vs missing
  • Clarifying questions — follow-ups for the submitter
  • Preliminary risk assessment — per-area ratings across 6 domains
  • Review summary — narrative for human reviewers
🔄

4. Submitter Revision

The submitter sees the AI feedback and can revise their answers, upload additional supporting documents (architecture diagrams, security assessments), and answer clarifying questions. They resubmit when ready.

👥

5. Human Review

RUAI reviewers receive a notification and evaluate the submission alongside the AI analysis. They work through a 12-item security checklist covering input validation, prompt hardening, output sanitization, logging, model isolation, rate limiting, least privilege, tool scoping, data validation, network segmentation, IAM, and human oversight. Reviewers can approve, reject, or request changes.

📚

6. Reference Docs

The Manage Docs library lets reviewers upload reference documents (policies, frameworks, guidelines). These are available for the AI reviewer to consult during analysis, ensuring evaluations align with organizational standards.

⚙️

LLM Configuration

Model: the LLM
Temperature: 0.2 (low randomness for consistent analysis)
Max tokens: 8,192
Response format: JSON enforced, validated against a Pydantic schema before storage
Execution: Background thread with live progress phases (connecting → analyzing → saving)
Re-run: Reviewers can re-trigger the AI review at any time

The Prompt — 7-Step Security Analysis

This is the exact instruction given to the LLM for each submission review.

1️⃣

Step 1 — Survey Interpretation

Extract and assess: use case description, LOB and business criticality, AI function and model types, data sensitivity, expected users.

2️⃣

Step 2 — Architecture Understanding

Identify: components, data flows, and trust boundaries from the submission. If architecture diagrams or DFDs were not provided, flag this as a gap.

3️⃣

Step 3 — AI-Specific Threat Surface

Evaluate each AI-specific threat against the use case:

  • Prompt Injection — Can user input manipulate AI behavior?
  • Data Poisoning — Could training/RAG data be tampered with?
  • Model Inversion — Could the model leak sensitive training data?
  • Hallucination Risks — Could AI generate false/misleading outputs with real-world consequences?
  • Model Misuse — Could the AI be used beyond its intended scope?
  • Autonomous Actions — Does the AI take actions without human approval?
4️⃣

Step 4 — Control Mapping Against 6 Threat Boundaries

For each boundary, evaluate whether the required security controls are Present, Partial, or Missing:

Boundary Key Risks Controls Required
UserPrompt Injection, ManipulationInput Validation, Prompt Hardening
ApplicationOutput Handling, Logging ExposureSanitization, Logging Controls
ModelModel Inversion, Data PoisoningModel Isolation, Rate Limiting
Tool / AgentAgentic Abuse, Privilege EscalationLeast Privilege, Tool Scoping
DataPoisoning, Supply Chain, Cross-TenantData Validation
InfrastructureDoS, IAM MisconfigurationSegmentation, Firewall
5️⃣

Step 5 — Threat Modelling & Risk Scoring

Identify threats → map each to controls → assign risk ratings with rationale.

6️⃣

Step 6 — Security Risk Assessment

Rate risk (Low / Medium / High / Critical) for each domain:

  • input_validation — Prompt injection defenses, input sanitization
  • data_protection — Data classification, encryption, access controls, cross-tenant isolation
  • model_security — Model isolation, rate limiting, inversion/poisoning defenses
  • output_handling — Output sanitization, hallucination safeguards, logging controls
  • access_control — IAM, least privilege, authentication, tool scoping
  • infrastructure — Network segmentation, firewall, DoS protection
7️⃣

Step 7 — Recommendations

Provide: specific mitigations for identified gaps, missing information to request, AI-specific safety questions.

📏

Risk Assessment Rubric

  • Critical — Prohibited use case, no human oversight on high-impact decisions, sensitive biometric data without controls, no input validation on user-facing AI
  • High — Impacts customers financially, autonomous AI actions without approval gates, limited access controls, personal data without classification
  • Medium — Drives decisions but with human review, vendor solution with limited transparency, partial security controls
  • Low — Advisory/informational only, full human oversight, no personal data, comprehensive security controls documented
