🚀 How the Detection-as-Code pipeline works

This treats detections the way we treat code: authored as portable Sigma, validated by an automated pipeline, and shipped to XSIAM through GitLab — with a human approving the merge. It's draft-and-approve: everything runs locally and read-only until you open the merge request.

1️⃣ Lint & schema-validate

The Sigma rule is parsed and checked for required structure (title, logsource, detection, condition) and best practices (severity, ATT&CK tags). Schema errors stop the pipeline.

2️⃣ Compile → XSIAM XQL

An in-house LLM (MetIQ GPT-4.1) translates the Sigma logic into an XSIAM XQL correlation query, mapping fields and noting any assumptions. A deterministic compiler is the floor if the model is busy.

3️⃣ Dry-run on the tenant

The compiled XQL is run read-only against the live XSIAM tenant over a bounded recent window — a real CI gate proving it parses, with the hit count over the last 24h.

4️⃣ Detection-engineering review

The LLM reviews quality, false-positive risk, ATT&CK coverage and gaps, suggests improvements, and flags overlap with detections already in the QRadar / CrowdStrike / Tanium catalog.

5️⃣ Package & merge

The pipeline emits the rule file, the compiled XQL, a .gitlab-ci.yml, and an MR description. Approve & open MR pushes it to the detection repo via the GitLab API — disabled on the dev instance (it always targets the prod repo). Every run and merge is logged.
