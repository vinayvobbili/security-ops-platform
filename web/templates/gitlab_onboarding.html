🧠 What is this?

It's a GitLab instance, hosted internally, opened up so anyone on the team can self-serve a repo without filing a ticket. The intent is "vibe coding" — quick experiments, scripts, glue code, half-baked ideas that don't need to live in a heavyweight platform.

Think of it as the citizen-developer sandbox: a place to try something, fork a starter, push a branch, and see if the idea has legs.

🌐
URL http://gitlab.internal.local — note this is plain HTTP, not HTTPS yet. Your browser will warn; that's expected for now.

1 Sign up

Go to the sign-up page:

http://gitlab.internal.local/users/sign_up

Fill in:

  • Your @the-company.com email address
  • A username
  • A password you pick yourself — you're not getting a temp password emailed to you
✅
That's it No admin approval. No email confirmation link. Submit the form and your account is active.

2 Sign in

Hit sign-in with the credentials you just set:

http://gitlab.internal.local/users/sign_in

You should land on your GitLab dashboard. If you don't see anything in the project list, that's normal — you haven't created or forked anything yet. On to step 3.

3 Fork the starter template

The vibe-lab group has a Python starter you can fork as a base for your own project:

http://gitlab.internal.local/vibe-lab/python-script

Open the link, click Fork (top-right of the project page), pick your namespace, and you'll have your own copy to push to.

🚀
Then what? Clone it locally, edit, commit, push. Standard git. If you've never used GitLab before, the project's own README in your fork has the next steps.

Cloning your fork

git clone http://gitlab.internal.local/<your-username>/python-script.git
cd python-script

4 Make your first push

Once your fork is cloned locally, change something small and push it back to confirm everything's wired up:

echo "# my notes" >> README.md
git add README.md
git commit -m "first commit"
git push origin main

The first git push will prompt for username + password. Username is your GitLab username; for the password, use a personal access token (not your account password) — see the gotcha below.

🔑 Generate a personal access token

  1. Sign into GitLab → click your avatar (top-right) → Edit profile.
  2. Left sidebar → Access Tokens.
  3. Name the token (e.g. laptop-cli), pick an expiry, and check at minimum the read_repository + write_repository scopes.
  4. Click Create and copy the token immediately — you can't view it again after closing the page.
  5. Paste that token when git prompts you for a password.
💾
Don't want to retype it every push? Run git config --global credential.helper store once. Git will remember the token after your next successful push. (Stored in plain text under ~/.git-credentials — fine for a personal laptop, not for shared machines.)

⚠️ Gotchas

🌐 Network & access

  • HTTP only for now. The site is http://, not https://. Your browser will flag the connection as insecure (no padlock, possibly a "Not Secure" warning) — that's expected. HTTPS rollout is on the roadmap.
  • Corp network required. Reachable from any corp-network path: office WiFi/LAN, corp VPN from home, or the corporate proxy. Off-network (coffee shop, hotel, no VPN) → connection just hangs.
  • No GitHub mirroring. Code pushed here lives here. There's no auto-sync to GitHub or any external service.

🔐 Auth & identity

  • Use a token, not your password, for git. GitLab disables password auth for git operations by default. If git push says HTTP Basic: Access denied, you used your password instead of a personal access token. See Step 4 above.
  • Forgot your password? Use the "Forgot your password?" link on the sign-in page — it emails a reset link to your @the-company.com address.
  • Cached the wrong credential? On Windows: Credential Manager → Windows Credentials → remove the gitlab.internal.local entry. On macOS/Linux with credential helper: git credential reject <<< "url=http://gitlab.internal.local" (or just clear ~/.git-credentials).
  • SSO / corp directory. This is local-account sign-up — your @the-company.com email is the identifier, but the password is separate from your corp AD password. Forgetting that is the #1 source of confusion.

🧱 Push & repo limits

  • Default branch is main. If you cloned a fork that still uses master, push to master or rename it locally.
  • No force-push to main. The default branch is protected on most projects. Push to a feature branch and open a merge request instead.
  • Big files. Large binaries (>10 MB) will work but get ugly fast. Use Git LFS if you really need to commit them.
🔒
Same data classification rules apply This is an internal tool, not a vault. Don't push secrets, credentials, customer PII, or anything you wouldn't put in a Confluence page. Treat it like any other corp code repo. If you push a secret by accident, rotate it and ask an admin to scrub the history — a force-push won't be enough.
